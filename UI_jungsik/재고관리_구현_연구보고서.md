## 1. 연구 배경 및 목적

### 1.1 연구 배경

현재 Farm-Manager 시스템은 수확 관리, 납품(B2B), 택배(B2C), 지출 관리, 인력 관리, 통합 결산 등 농장 운영의 핵심 기능을 충실히 구현하고 있다. 그러나 수확과 판매 사이의 **재고(在庫)** 개념이 존재하지 않아 다음과 같은 한계가 있다.

- 수확한 작물이 실제로 얼마나 남아있는지 알 수 없음
- 납품/택배 시 재고 초과 판매가 발생해도 시스템이 경고하지 않음
- 폐기·손실된 물량이 통계에서 누락됨
- 실질 수익 계산 시 손실분이 반영되지 않음

농장주 입장에서는 "재고관리가 없어도 운영에 큰 문제없다"는 현실적 판단도 유효하다. 딸기·채소류 등 신선 농산물은 보관 기간이 짧아 전통적 재고관리 개념과 다르기 때문이다.

따라서 본 연구는 **"재고관리를 사용할지 말지 농장주가 직접 선택할 수 있는 시스템"** 을 전제로, 구현 방안 3가지를 제시하고 최적안을 도출한다.

### 1.2 연구 목적

1. 농업 환경에 적합한 재고관리 모델 정의
2. 기존 시스템과 자연스럽게 통합되는 3가지 구현안 제시
3. 설정(Settings)에서 ON/OFF 가능한 토글 방식 설계
4. 최종 선택안의 구체적 구현 계획 제시

### 1.3 연구 범위

- 대상 작물: 딸기 등 신선 원물 + 가공품 모두 포함
- 재고 단위: kg, 박스, 개 등 Farm-Manager 기존 단위 체계 사용
- 통합 대상: 수확(Harvest), 납품(Bulk), 택배(Courier), 통합결산(Finance)

## 2. 현재 시스템 현황 분석

### 2.1 현재 데이터 흐름

```
[수확 기록]
  └─ harvest_records 테이블
       - farm_id, house_number, grade, quantity, crop_name
       - recorded_at

          ↓ (연결 없음 — 재고 개념 부재)

[납품 기록]            [택배 기록]
  └─ sales_records      └─ sales_records
       - sale_type: b2b       - sale_type: b2c
       - quantity             - quantity
       - price                - price
```

현재 수확 기록과 판매 기록 사이에 **재고 검증 로직이 없다.** 즉 수확 0kg 상태에서도 납품 100kg 등록이 가능하다.

### 2.2 현재 통합결산(Finance) 페이지의 재고 관련 표시

통합결산 페이지에서는 다음 수치를 계산한다:

| 항목 | 계산 방법 | 재고 반영 여부 |
|------|-----------|---------------|
| 총 수확량 | SUM(harvest_records.quantity) | - |
| 총 판매량 | SUM(sales_records.quantity) | - |
| 판매 차이 | 수확량 - 판매량 | 단순 차이, 재고 아님 |
| 미정산 금액 | 납품 중 미정산 합계 | 금액만 |

### 2.3 재고관리 미구현으로 인한 실제 문제

**사례 1: 과잉 판매**
- 수확: 300kg
- 납품 등록: 200kg
- 택배 등록: 150kg
- 실제 판매: 350kg (수확보다 50kg 초과)
- 현재 시스템: 경고 없이 등록 허용

**사례 2: 폐기 손실 미반영**
- 수확: 500kg
- 납품+택배: 400kg
- 폐기(상품가치 없음): 80kg
- 잔여 재고: 20kg이어야 하지만 시스템상 100kg으로 표시됨

**사례 3: 등급별 재고 불일치**
- 상(上) 등급: 납품용
- 중(中) 등급: 택배용
- 하(下) 등급: 폐기 또는 저가 처리
- 현재 등급별 재고 추적 불가

### 2.4 현재 시스템으로도 충분한 케이스

딸기 농장의 특성상 수확 당일 또는 익일 전량 출하하는 경우가 많다. 이 경우:
- 재고가 실질적으로 하루를 넘기지 않음
- 재고 추적의 실익이 적음
- 시스템 복잡도만 증가할 수 있음

**→ 따라서 재고관리는 선택(Optional) 기능으로 구현하는 것이 타당하다.**

## 3. 농업 재고관리의 특수성

### 3.1 일반 재고관리 vs 농업 재고관리

| 구분 | 일반 재고관리 | 농업 재고관리 |
|------|-------------|-------------|
| 입고 | 구매/제조 | 수확 (자연 생산) |
| 출고 | 판매 | 납품 + 택배 + 폐기 |
| 유통기한 | 길거나 없음 | 매우 짧음 (1~3일) |
| 단위 | 고정 | 유동적 (kg, 박스, 개) |
| 손실 | 보관 손상 | 자연 부패, 등급 하락 |
| 재고 평가 | FIFO, LIFO | 당일 기준 |

### 3.2 딸기 농장 재고의 특성

**생리적 특성:**
- 수확 후 상온 2~4시간, 냉장 3~5일 보관 가능
- 수확 즉시 선별 → 포장 → 출하 사이클이 빠름
- 등급(상/중/하)에 따라 출하 경로가 다름

**운영적 특성:**
- 수확량이 날씨/온도에 따라 변동 폭이 큼
- 수확 후 폐기율이 10~30%에 달할 수 있음
- 선물세트 등 가공 시 단위 변환 발생

### 3.3 재고 손실 유형 분류

```
수확 입고
    │
    ├─ [등급 선별 손실] 선별 과정에서 제외된 물량
    ├─ [포장 손실] 포장 중 파손
    ├─ [보관 손실] 냉장 보관 중 부패
    ├─ [운반 손실] 운반 중 파손
    └─ [실질 출하]
           ├─ 납품(B2B)
           └─ 택배(B2C)
```

### 3.4 농업 재고관리 구현 시 고려사항

1. **단순성 우선:** 복잡한 ERP 방식은 현장에서 사용 불가
2. **모바일 최적화:** 밭/창고에서 스마트폰으로 입력
3. **빠른 입력:** 수확 직후 빠르게 등록 가능해야 함
4. **폐기 처리 용이성:** 폐기 등록이 번거로우면 안 씀
5. **기존 데이터와 연속성:** 현재까지의 데이터 유지

## 4. 재고관리 ON/OFF 토글 설계 철학

### 4.1 토글 위치

**Settings(설정) 페이지 → 농장 기능 설정 섹션:**

```
┌─────────────────────────────────────┐
│ 재고관리 사용                        │
│ 수확 재고를 기반으로 납품/택배       │
│ 가능 수량을 제한합니다.              │
│                          [OFF ◯] │
└─────────────────────────────────────┘
```

### 4.2 OFF 상태 (기본값)

- **현재 시스템 그대로 유지**
- 수확과 판매가 독립적으로 동작
- 재고 관련 UI 일체 숨김
- 기존 사용자의 워크플로우 변화 없음

### 4.3 ON 상태

- **재고 잔량 계산 활성화**
- 납품/택배 등록 시 잔여 재고 확인
- 잔여 재고 초과 시 경고 표시 (강제 차단 or 경고만)
- 폐기 등록 기능 활성화
- 통합결산에 재고 현황 섹션 추가

### 4.4 토글 전환 시 데이터 처리

| 전환 방향 | 기존 데이터 처리 |
|----------|----------------|
| OFF → ON | 기존 수확/판매 기록 기반으로 초기 재고 자동 계산 |
| ON → OFF | 재고 데이터는 보존, 단 UI에서만 숨김 |

### 4.5 Supabase farms 테이블 변경

```sql
ALTER TABLE farms ADD COLUMN inventory_enabled BOOLEAN DEFAULT FALSE;
```

## 5-1. 1안: 단순 수확 연동형 재고관리

### 개요

수확 기록을 자동으로 재고 입고로 처리하고, 납품/택배 등록 시 자동으로 재고 차감하는 가장 단순한 방식.

### 핵심 원리

```
수확 기록 등록
    → 재고 자동 +입고 (수확량만큼)

폐기 등록
    → 재고 -차감

납품 등록
    → 잔여 재고 확인
    → 재고 -차감

택배 등록
    → 잔여 재고 확인
    → 재고 -차감
```

### 데이터 구조

```sql
-- 재고 현황 뷰 (별도 테이블 불필요)
-- 수확 - 판매 - 폐기 = 잔여 재고

CREATE TABLE inventory_adjustments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    farm_id UUID REFERENCES farms(id),
    crop_name TEXT NOT NULL,
    adjustment_type TEXT NOT NULL, -- 'waste' | 'correction'
    quantity NUMERIC NOT NULL,     -- 폐기 시 음수
    reason TEXT,
    adjusted_at TIMESTAMPTZ DEFAULT NOW()
);

-- 잔여 재고 계산 (View)
-- = SUM(harvest) - SUM(sales) - SUM(waste)
```

### UI 변경사항

**납품/택배 등록 폼:**
```
품목: 딸기
수량 입력: [____] kg
잔여 재고: 85kg 남음  ← 새로 추가
[등록하기]
```

**수확 관리 페이지에 재고 현황 배지 추가:**
```
오늘 수확  280kg
현재 잔고  ← 배지 (초록: 충분 / 빨강: 부족)
  상등급  120kg
  중등급   45kg
  하등급   15kg
```

**폐기 등록 버튼 (수확 관리 페이지):**
```
[+ 폐기 등록]
품목: 딸기
수량: 30kg
사유: 부패 / 파손 / 등급 외
```

### 장점

- 구현이 단순 (기존 테이블 거의 변경 없음)
- 이해하기 쉬움 ("수확 = 입고, 판매 = 출고")
- 모바일에서 추가 작업 최소화
- 기존 데이터와 완벽 호환
- 개발 기간: 약 3~5일

### 단점

- 창고 재고와 밭 재고를 구분 못함
- 선별/포장 과정의 손실을 세분화 못함
- 등급별 재고가 수확 등급과 연동되므로 후처리 시 오차 가능

### 적합한 농장 유형

- 수확 즉시 당일 출하하는 농장
- 재고관리 처음 도입하는 농장
- 폐기율이 낮은 농장

## 5-2. 2안: 독립 창고 재고관리

### 개요

수확과 판매 사이에 **창고(Warehouse)** 개념을 독립적으로 추가. 수확물이 창고에 입고된 후, 창고에서 납품/택배로 출고되는 2단계 구조.

### 핵심 원리

```
[수확]                    [창고]                    [판매]
수확 기록                창고 입고 등록              납품/택배 등록
(자동 → 창고 입고 제안)  (수확량 연동 or 수동)       (창고 잔고 차감)
```

### 데이터 구조

```sql
-- 창고 입출고 테이블
CREATE TABLE warehouse_stock (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    farm_id UUID REFERENCES farms(id),
    crop_name TEXT NOT NULL,
    grade TEXT,                     -- 상/중/하
    quantity NUMERIC NOT NULL,
    transaction_type TEXT NOT NULL, -- 'in' | 'out' | 'waste' | 'correction'
    reference_type TEXT,            -- 'harvest' | 'sale' | 'manual'
    reference_id UUID,              -- harvest_id or sale_id
    notes TEXT,
    transacted_at TIMESTAMPTZ DEFAULT NOW()
);

-- 현재 재고 = SUM(in transactions) - SUM(out transactions)
```

### UI 변경사항

**창고 현황 전용 페이지 신설 (`/warehouse`):**
```
┌──────────────────────────────┐
│ 창고 현황                     │
│                              │
│ 딸기                          │
│  상등급  ████░░ 120kg / 200kg │
│  중등급  ██░░░░  45kg / 200kg │
│  하등급  █░░░░░  15kg / 200kg │
│                              │
│ [입고 등록]  [폐기 등록]       │
└──────────────────────────────┘
```

**수확 기록 등록 후 팝업:**
```
수확 기록이 저장되었습니다.
창고에 입고 하시겠습니까?
[창고 입고 등록] [나중에]
```

### 장점

- 수확과 창고 재고를 명확히 분리
- 창고 입고 시점을 수동으로 제어 가능
- 부분 입고 처리 가능 (수확 300kg 중 200kg만 입고)
- 창고별 관리 가능성 (미래 확장)
- 실제 물류 흐름을 가장 잘 반영

### 단점

- **추가 작업 증가:** 수확 등록 후 창고 입고를 별도로 해야 함
- **입고를 빠뜨릴 위험:** 창고 입고를 안 하면 재고 = 0 상태에서 납품 불가
- **모바일 환경에서 2단계 입력 부담**
- 개발 기간: 약 7~10일
- 신규 페이지(/warehouse) 추가 필요

### 적합한 농장 유형

- 창고가 있고 보관 기간이 긴 작물 (감자, 고구마, 양파 등)
- 수확에서 출하까지 시간 간격이 있는 농장
- 재고 추적을 정밀하게 하고 싶은 농장

## 5-3. 3안: 하이브리드 단계별 추적형

### 개요

수확 → 선별 → 포장 → 창고 → 출하의 전 과정을 추적하는 가장 정밀한 방식. 제조업 수준의 재고관리를 농장에 적용.

### 핵심 원리

```
1단계: 수확
   └─ 총 수확량 기록 (등급 미분류)

2단계: 선별
   └─ 등급별 분류 (상/중/하)
   └─ 선별 손실량 기록

3단계: 포장
   └─ 박스/kg 단위 포장
   └─ 포장 손실 기록

4단계: 창고 입고
   └─ 포장 완료된 수량 창고 등록

5단계: 출하
   └─ 납품 or 택배로 출고
   └─ 창고 자동 차감
```

### 데이터 구조

```sql
-- 재고 프로세스 단계 테이블
CREATE TABLE inventory_process (
    id UUID PRIMARY KEY,
    farm_id UUID,
    crop_name TEXT,
    process_date DATE,

    -- 수확 단계
    harvested_quantity NUMERIC,

    -- 선별 단계
    grade_sang NUMERIC,      -- 상등급 분류량
    grade_jung NUMERIC,      -- 중등급 분류량
    grade_ha NUMERIC,        -- 하등급 분류량
    sort_loss NUMERIC,       -- 선별 손실

    -- 포장 단계
    packed_sang NUMERIC,     -- 포장 완료 (상)
    packed_jung NUMERIC,     -- 포장 완료 (중)
    pack_loss NUMERIC,       -- 포장 손실

    -- 상태
    process_status TEXT,     -- 'harvested' | 'sorted' | 'packed' | 'stored'
    notes TEXT,
    created_at TIMESTAMPTZ
);
```

### 장점

- 가장 정밀한 재고 추적
- 단계별 손실 원인 파악 가능

## 7. 최종 선택: 1안 — 단순 수확 연동형 상세 구현 계획

### 7.1 선택 근거

1. **현실성:** 딸기 농장은 수확 즉시 출하. 복잡한 창고 프로세스 불필요
2. **모바일 우선:** 밭에서 스마트폰으로 빠르게 등록하는 환경
3. **도입 마찰 최소화:** 기존 작업 흐름 변화 없이 재고만 추가
4. **확장 가능:** 1안 안정화 후 2안으로 업그레이드 용이
5. **빠른 개발:** 3~5일 내 완성 가능

### 7.2 재고 계산 공식

```
현재 재고 =
  SUM(harvest_records.quantity)          ← 수확 입고
  - SUM(sales_records.quantity)          ← 납품+택배 출고
  - SUM(inventory_adjustments.quantity)  ← 폐기/보정
```

### 7.3 재고 경고 레벨 정의

| 상태 | 조건 | 색상 | 동작 |
|------|------|------|------|
| 충분 | 잔고 > 요청량 × 1.2 | 초록 | 정상 등록 |
| 주의 | 잔고 > 요청량 | 노랑 | 경고 표시 후 등록 가능 |
| 부족 | 잔고 < 요청량 | 빨강 | 경고 표시 (강제 차단 아님) |
| 재고 없음 | 잔고 ≤ 0 | 빨강 | 강한 경고 (등록은 가능) |

> **중요:** 재고 부족 시 강제 차단하지 않는다.
> 현장에서 급하게 등록해야 하는 상황을 막으면 안 된다.
> 경고(Warning) 방식으로 알리되 등록은 허용한다.

## 8. 데이터베이스 설계

### 8.1 신규 테이블: inventory_adjustments

```sql
CREATE TABLE inventory_adjustments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    farm_id UUID NOT NULL REFERENCES farms(id) ON DELETE CASCADE,
    crop_name TEXT NOT NULL,
    grade TEXT,                        -- NULL이면 전체 등급
    adjustment_type TEXT NOT NULL
        CHECK (adjustment_type IN ('waste', 'correction', 'opening_stock')),
    quantity NUMERIC(10,2) NOT NULL,   -- 폐기: 양수(줄어드는 양), 보정: 음수 가능
    reason TEXT,                       -- 폐기 사유 (부패/파손/등급외/기타)
    adjusted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS 정책
ALTER TABLE inventory_adjustments ENABLE ROW LEVEL SECURITY;
CREATE POLICY "farm_owner_only" ON inventory_adjustments
    USING (farm_id IN (
        SELECT id FROM farms WHERE owner_id = auth.uid()
    ));
```

### 8.2 farms 테이블 컬럼 추가

```sql
ALTER TABLE farms
ADD COLUMN inventory_enabled BOOLEAN NOT NULL DEFAULT FALSE,
ADD COLUMN inventory_warn_only BOOLEAN NOT NULL DEFAULT TRUE;
-- warn_only: true=경고만, false=강제차단
```

### 8.3 재고 현황 조회 쿼리

```sql
-- 작물별 현재 재고 조회
SELECT
    crop_name,
    SUM(harvested) - SUM(sold) - SUM(wasted) AS current_stock
FROM (
    -- 수확 입고
    SELECT crop_name, quantity AS harvested, 0 AS sold, 0 AS wasted
    FROM harvest_records
    WHERE farm_id = $1
    AND recorded_at >= CURRENT_DATE - INTERVAL '30 days'  -- 30일 이내

    UNION ALL

    -- 판매 출고
    SELECT crop_name, 0, quantity AS sold, 0
    FROM sales_records
    WHERE farm_id = $1
    AND sale_type IN ('b2b', 'b2c')
    AND recorded_at >= CURRENT_DATE - INTERVAL '30 days'

    UNION ALL

    -- 폐기/보정
    SELECT crop_name, 0, 0, quantity AS wasted
    FROM inventory_adjustments
    WHERE farm_id = $1
    AND adjusted_at >= CURRENT_DATE - INTERVAL '30 days'
) sub
GROUP BY crop_name;
```

### 8.4 재고 초기화 (ON 전환 시)

```sql
-- 재고관리 ON 전환 시 기존 데이터 기반 초기 재고 계산
-- (기존 수확 - 기존 판매로 자동 계산됨, 별도 입력 불필요)
UPDATE farms SET inventory_enabled = TRUE WHERE id = $1;
```

## 11. 결론 및 권고사항

### 11.1 최종 권고: 1안 채택

비키트 에이전트 팀은 현재 Farm-Manager의 사용 환경과 딸기 농장의 특성을 종합 검토한 결과, **1안(단순 수확 연동형)**을 강력히 권고한다.

**핵심 이유:**
- 딸기는 당일~익일 출하가 기본, 복잡한 창고 관리 불필요
- 모바일 우선 환경에서 추가 입력 단계는 사용자 이탈 유발
- 기존 운영 데이터와 완전 호환
- 개발 빠르고 안정적

### 11.2 재고관리 OFF가 기본값인 이유

1. **현재 시스템도 거의 완성형:** 재고가 없어도 수확·판매·결산이 모두 가능
2. **농장주 선택권 보장:** 재고관리가 필요없는 농장에 강제 적용 불필요
3. **도입 리스크 제로:** OFF 상태에서는 현재 워크플로우 완전 유지

### 11.3 재고관리 ON 상태의 기대 효과

| 기대 효과 | 설명 |
|----------|------|
| 과잉 판매 방지 | 수확보다 많이 판매하는 오류 사전 예방 |
| 폐기 추적 | 월별 폐기율 파악으로 생산 효율 개선 |
| 경영 인사이트 | 실질 생산원가 계산 가능 |
| 재고 분쟁 방지 | "얼마나 남았지?" 확인 가능 |

### 11.4 미래 확장 계획 (선택 사항)

```
현재: 1안 (단순 연동)
   ↓ (안정화 후)
미래: 1.5안 (창고 선택 추가)
   - 수확 → 즉시 출하 or 창고 입고 선택
   - 창고 입고된 것만 납품/택배 가능하도록 옵션 추가
```

### 11.5 즉시 구현 여부 결정

본 연구보고서를 바탕으로 농장주께서 결정:

```
A. 지금 구현: "재고관리 구현 시작해줘" → 바로 착수
B. 보류: "나중에 구현" → 현재 시스템 그대로 운영
C. 현재 시스템 완성: "재고관리 불필요" → OFF 기본값으로 최종 배포
```
